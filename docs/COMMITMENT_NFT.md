# Commitment NFT Contract — Integrator Notes

This page documents the `commitment_nft` contract behavior, trust boundaries, validation rules, and testing notes for integrators and reviewers.

## Summary

The `commitment_nft` contract mints and manages commitment NFTs that represent locked commitments with metadata including duration, max-loss, and type. Commitment IDs are auto-generated by the contract in the format `COMMIT_{token_id}`.

## Public APIs (high level)

- `initialize(admin: Address) -> Result<(), ContractError>`
  - Admin-only initialization. Sets `Admin`, `TokenCounter`, `TokenIds` and pause state.
  - Errors: `AlreadyInitialized` if called twice.

- `set_core_contract(core_contract: Address) -> Result<(), ContractError>`
  - Admin-only. Stores the authorized `core_contract` address used for settlement operations.

- `add_authorized_contract(caller: Address, contract_address: Address) -> Result<(), ContractError>`
  - Admin-only. Adds a minter to the whitelist so it may call `mint`.

- `mint(caller: Address, owner: Address, _commitment_id: String, duration_days: u32, max_loss_percent: u32, commitment_type: String, initial_amount: i128, asset_address: Address, early_exit_penalty: u32) -> Result<u32, ContractError>`
  - Creates a new `CommitmentNFT` and returns the minted `token_id`.
  - Notes: user-supplied `commitment_id` is ignored; the contract auto-generates `COMMIT_{token_id}` and indexes it for reverse lookup.

- `get_metadata(token_id: u32) -> Result<CommitmentNFT, ContractError>`
  - Returns the stored `CommitmentNFT`.

- `get_commitment_by_id(commitment_id: String) -> Result<CommitmentNFT, ContractError>`
  - Reverse lookup via `commitment_id -> token_id` then returns the NFT.

## Validation rules and parameter caps

- `duration_days` must be > 0. Otherwise `InvalidDuration`.
- `max_loss_percent` must be <= 100. Otherwise `InvalidMaxLoss`.
- `commitment_type` must be one of: `"safe"`, `"balanced"`, `"aggressive"`. Otherwise `InvalidCommitmentType`.
- `initial_amount` must be > 0 (signed i128). Otherwise `InvalidAmount`.
- `owner` must not be the "zero" Stellar address (contract uses canonical zero address string). Otherwise `TransferToZeroAddress`.
- Expiration timestamp calculation checks for overflow and returns `ExpirationOverflow` if duration math overflows u64.

Note: There is no global supply cap implemented in the contract. If a supply or per-owner cap is required, the integrator should request an enhancement; tests will be added to reflect the chosen policy once implemented.

## Trust boundaries and auth

- Admin (stored at `DataKey::Admin`) has authority to:
  - Pause/unpause
  - Set core contract
  - Add/remove authorized minters
  - Upgrade/migrate/enter emergency mode
- Authorized minters: either admin, the `CoreContract` address, or addresses added via `add_authorized_contract` may call `mint`.
- The contract enforces `admin.require_auth()` and `caller.require_auth()` where appropriate. Calls from other contracts require the calling contract to be present in the whitelist.

## Reentrancy and safety

- A simple boolean `ReentrancyGuard` stored in instance storage prevents reentrant execution: functions set it to `true` at entry and clear it before returning. If it's already `true`, functions return `ReentrancyDetected`.
- The contract calls `EmergencyControl::require_not_emergency(&e)` and `Pausable::require_not_paused(&e)` where appropriate to respect higher-level controls.
- Arithmetic checks: timestamp math uses `checked_mul` and `checked_add` to avoid overflow and returns `ExpirationOverflow` on failure.

## Error codes of interest

See `ContractError` enum in `contracts/commitment_nft/src/lib.rs` for full list. Key error codes for mint validation:

- `NotInitialized` = 1
- `AlreadyInitialized` = 2
- `NotAuthorized` = 6
- `InvalidDuration` = 10
- `InvalidMaxLoss` = 11
- `InvalidCommitmentType` = 12
- `InvalidAmount` = 13
- `ReentrancyDetected` = 14
- `ExpirationOverflow` = 20
- `TransferToZeroAddress` = 18

## Testing notes

- Unit tests live in `contracts/commitment_nft/src/tests.rs` and use the Soroban host `Env` test harness.
- Test runner for wasm-target tests (recommended):

```bash
rustup target add wasm32v1-none
cargo test -p commitment_nft --target wasm32v1-none --release
```

- If the wasm target is not installed, tests will fail to compile (error: can't find crate for `core`).
- The test suite includes checks for parameter validation, events, reverse lookup, and reentrancy guard behavior.

## Integration recommendation

- Validate that callers setting `CoreContract` or adding authorized minters are running with appropriate governance.
- Consider adding a supply cap or per-owner caps if business rules require limiting minted commitments. If added, enforce in `mint` with admin-settable `MaxSupply` and include tests for `MaxSupply` edge cases.
